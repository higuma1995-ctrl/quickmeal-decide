# requirements.md
_最終更新：2026-04-14_

## 1. 目的

「何食べる？」の答えは大体わかっている。でも決め手がない・言い出しにくい・考えるのがめんどくさい。
本アプリはその最後の一押しと免責を与え、食事選択の摩擦をゼロにする。
判断の最終責任は人間が持ち、本アプリは意思決定に伴う摩擦だけを最小化する。

## 2. 収益モデル

- 再抽選2回目以降に擬似広告視聴（5秒カウント）を挟む
- 将来的なGoogle AdSense差し替えを前提とした構造で実装する
- 広告視聴率を下げない設計（視聴しないと再抽選できない導線を維持する）

## 3. 対象ユーザーと使い方

ライト層：開いてすぐ「決める」を押す。フィルタは触らない
複数人・デート：一人が操作→結果を見せる、またはLINEで共有
ヘビー層：候補・タグをカスタマイズして精度を上げる

## 4. 利用シーン

- 会議後・飲み前・移動中など、その場で即決が必要なとき
- 複数人で「これでどう？」と共有する前段
- 一人でランチ・ディナーを決めるとき
- 二次会・カフェ・締めなど場面ごとの絞り込み

## 5. スコープ

### 5-1. 候補
- 候補は「名前」を持つ
- プリセット候補を20〜30件用意（定番のみ）
- ユーザーは候補を自由に追加・完全削除できる
- プリセット候補は完全削除不可・抽選からの除外のみ可能
- 各候補にタグを設定できる（プリセットの初期タグは編集可能）

### 5-2. タグ
予算：節約 / 定番 / ご馳走
時間帯：ランチ / ディナー
シーン：カフェ / 食事メイン / 飲み会 / 二次会

- タグは複数設定可能
- タグ未設定の候補はどのフィルタ条件でも除外されない

### 5-3. フィルタ
- 完全任意（使わなくても成立）
- 同一軸内：OR条件
- 異なる軸間：AND条件

### 5-4. 抽選・決定
- 「決める」で候補を1つ表示
- 結果表示は候補名のみ（理由・一言は表示しない）
- 結果が表示された時点でログに自動保存（確定操作不要）
- 却下操作あり（セッション中の一時除外）。セッション終了でリセット
- 再抽選：セッション開始時に無料回数1回のみ付与。2回目以降は擬似広告視聴（5秒）が必要
- 候補枯渇時は一時除外をリセット
- 直近5件のログに出た候補は抽選で出にくくする（重み0.3）
- フィルタ絞り込みにより候補が0件の場合は「条件に合う候補がありません。フィルタを変えてみてください。」を表示し、決めるボタンは非活性

### 5-5. ログ
- 決定した候補を日付・候補名で保存
- 個別削除・全消去の両方可能
- 編集不可
- ログは抽選の重み付けに使用する

### 5-6. 共有
- 決定結果をワンタップでテキストコピー
- 固定テンプレート：「今日のごはんは【○○】に決まりました🍽️」

### 5-7. チュートリアル
- 初回起動時のみモーダルで表示
- 内容：「迷ったらそのまま『決める』を押してください」
- 「閉じる」ボタンで消える

### 5-8. 画面構成
- ボトムナビゲーション（タブバー）構成
- タブ①「決める」：フィルタ＋決めるボタン＋結果表示
- タブ②「候補」：追加・削除・タグ編集・除外管理
- タブ③「ログ」：履歴表示・個別削除・全消去

## 6. やらないこと
- 外部API使用
- ログイン・同期機能
- AIレコメンド・ランキング・評価
- 位置情報・時刻自動判定
- 人数タグ（廃止）
- 空気モード（廃止）
- キャラクター・一言コメント

## 7. 成功条件
- 開いて3秒以内に「決める」が押せる状態を作れること
- ライト層がフィルタなしで違和感なく使えること
- ヘビー層が候補・タグをカスタマイズして精度を上げられること
- 再抽選導線で広告視聴が自然に発生すること
- 将来的にAdSenseへ差し替えられる構造であること
- spec.mdとコードが完全一致していること
- GitHub Pagesで公開済みであること
